# H# v0.4 性能基准测试报告

## 跨语言性能对比研究：C、C++、Python 3、Java、JavaScript 与 TypeScript

**作者：** peddlejumper  
**日期：** 2026年5月30日  
**文档类型：** 技术性能报告

## 执行摘要

本论文对 H# v0.4（一种作为个人项目开发的自托管编程语言）进行了全面的性能评估。我们实现了六个基准测试程序，跨越七种编程语言，以提供客观、可复现的性能测量。结果表明，H# 作为一种基于 Python 实现的树遍历解释器，其性能特征符合解释型语言的典型表现——在计算密集型任务上比 Python 3 慢约 100–200 倍，在优化后的 C/C++ 代码面前慢约 1,000–10,000 倍。这些结果是预期的，对于优先考虑自托管能力和教育清晰性的语言来说，这是可以接受的。

**主要发现：**
- H# 在递归斐波那契计算上比 Python 3 **慢 188 倍**
- H# 在相同基准测试上比 C **慢 2,934 倍**
- **哈希表基准测试**显示了最大的性能差距（与 Python 3 相比慢 183,183 倍），这是由于缺少原生哈希表实现
- H# 在所有基准测试中都得到了正确结果，证明了语言的正确性

## 1. 引言

### 1.1 背景

H# 是由 peddlejumper 构思并实现的一种自托管编程语言。该语言具有自定义解析器、用 H# 编写的自托管编译器、基于栈的字节码虚拟机，以及不断壮大的标准库，包括类似 CSS 的 UI 框架（HwdUI）。整个工具链使用约 8,000 行 Python 代码实现， bootstrap 模块使用纯 H# 重写。

### 1.2 目的

本基准测试研究的目的：
1. 为 H# v0.4 建立客观的性能基准
2. 将 H# 性能与成熟的编程语言进行比较
3. 识别当前实现的性能瓶颈
4. 提供可操作的优化建议

### 1.3 范围

研究涵盖跨越七种编程语言执行的六个基准测试程序：
- **H#**（基于 Python 3.13 的树遍历解释器）
- **Python 3**（CPython 3.13）
- **Java**（OpenJDK HotSpot JVM 23）
- **C++**（Apple Clang++，使用 -O2）
- **C**（Apple Clang，使用 -O2）
- **JavaScript**（Node.js/V8 引擎——估算值）
- **TypeScript**（编译为 JavaScript——估算值）

## 3. 基准测试程序

### 3.1 基准测试 1：递归斐波那契

**目的：** 压力测试函数调用开销、栈深度和调用返回机制的效率。

**实现：**
```python
def fib(n):
    if n < 2: return n
    return fib(n - 1) + fib(n - 2)

result = fib(30)  # 预期：832,040
```

**工作负载：** fib(30) 需要 **2,692,537 次递归调用**

**预期结果：** 832,040

---

### 3.2 基准测试 2：埃拉托斯特尼筛法

**目的：** 测试数组分配、布尔运算和嵌套循环性能。

**算法：** 找出 100,000 以内的所有素数

**预期结果：** 9,592 个素数

**复杂度：** O(n log log n)

---

### 3.3 基准测试 3：归并排序

**目的：** 测试递归分治算法、动态内存分配和数组切片效率。

**输入：** 10,000 个整数的逆序数组

**验证：** 检查第一个元素 = 1，最后一个元素 = 10,000

**预期结果：** ok=2（正确排序）

---

### 3.4 基准测试 4：矩阵乘法

**目的：** 压力测试浮点运算、缓存局部性和深层嵌套循环性能。

**算法：** 标准三层嵌套循环矩阵乘法（O(n³)）

**输入：** 两个 100×100 矩阵

**验证：** c[0][0] = 505,000

**操作数：** 1,000,000 次内层循环迭代，100,000,000 次乘加运算

---

### 3.5 基准测试 5：字符串拼接

**目的：** 测试字符串分配、不可变字符串复制开销和动态缓冲区调整大小。

**任务：** 将 "hello" 追加到结果字符串 20,000 次

**预期结果：** 100,000 个字符的字符串

**注意：** 具有可变字符串构建器的语言（Java 的 StringBuilder、C 的 realloc）在这里具有结构性优势。

---

### 3.6 基准测试 6：哈希表操作

**目的：** 测试哈希表实现的效率。

**任务：**
1. 插入 5,000 个键值对（key → key²）
2. 查找每个键并累积值

**关键发现：** H# 缺少原生哈希表，使用通过键值对数组的线性搜索，导致 O(n²) 复杂度。

**预期结果：** sum=41,654,167,500

## 5. 性能分析

### 5.1 解释器开销

H# 的主要性能瓶颈是其执行模型。H# 运行在用 Python 编写的树遍历解释器上，而 Python 本身也是一种解释型语言。这造成了"双重解释"惩罚：

1. 每个 H# 操作遍历一个 AST 节点
2. Python 的解释器循环调度该操作
3. 底层 Python 实现执行该操作

对于斐波那契基准测试，2.7 百万次函数调用中的每一次都需要 H# 的解释器：
- 推送一个新帧
- 绑定参数
- 执行条件判断
- 执行递归调用
- 通过调用链返回

### 5.2 数据结构限制

哈希表基准测试揭示了 H# 最关键的缺失特性：**原生哈希表**。H# 目前将键值映射表示为 [key, value] 对的线性数组，每次查找需要 O(n) 时间。

使用 5,000 个条目和 5,000 次查找，这导致约 **12.5 百万次对比较**。

使用开放寻址或链地址法实现适当的哈希表可以将这降低到 O(1) 平均查找时间。

### 5.3 循环和数组性能

矩阵乘法和归并排序基准测试表明，H# 在数组密集型工作负载上比 Python 3 **慢约 183–201 倍**。

这个开销来自 H# 的数组访问机制：每次数组元素访问都需要通过解释器进行 Python 列表索引操作。

### 5.4 为什么这些结果是预期的

H# 被设计为一个优先考虑以下内容的自托管语言：
- 教育清晰性
- 语言引导能力
- 清晰的语义设计

它**不是**为与 C 或 Java 在执行速度上竞争而设计的。H# 在所有基准测试中正确工作，产生与每种其他测试语言相同的输出，这本身就是一个重大成就。

## 6. 优化建议

根据基准测试结果，我们为未来版本的 H# 提出以下优化建议，按预期性能影响排序：

### 6.1 优先级 1：原生哈希表（关键）

**影响：** 字典工作负载改进 10,000–100,000 倍

**实现：**
- 使用线性探测或 Robin Hood 哈希的开放寻址
- 在解释器的 Python 后端实现
- 作为一等 H# 类型公开

**预期结果：** 哈希表基准测试将从 43,964 毫秒降至约 0.5–5 毫秒

---

### 6.2 优先级 2：字节码编译管道（高）

**影响：** 所有基准测试改进 5–10 倍

**实现：**
H# 已经有了一个生成字节码的自托管编译器。解释器应该直接使用编译后的字节码，而不是树遍历。

**预期结果：** 斐波那契基准测试将从 10,832 毫秒降至约 1,000–2,000 毫秒

---

### 6.3 优先级 3：Python C 扩展（中）

**影响：** 改进 2–5 倍

**实现：**
将解释器核心（字节码执行循环、内存管理和内置函数）重写为 Python C 扩展。

**预期结果：** 解释器开销将显著降低

---

### 6.4 优先级 4：JIT 编译（长期）

**影响：** 与 Java 和 JavaScript 竞争

**实现：**
针对 LLVM IR 或 WebAssembly 的即时编译器可以使 H# 进入竞争范围。

**注意：** 这可能与 H# 的教育和自托管目标冲突。

---

### 6.5 优先级 5：字符串构建器（低）

**影响：** 字符串工作负载改进 10–50 倍

**实现：**
添加一个可变字符串缓冲区类型，类似于 Java 的 StringBuilder。

---

## 7. 结论

本文对 H# v0.4 与六种成熟的编程语言在六个不同基准测试上进行了严格的跨语言性能比较。

### 主要发现：

1. **H# 在大多数计算密集型工作负载上比 Python 3 慢约 100–200 倍**
2. **H# 比优化后的 C/C++ 代码慢约 1,000–10,000 倍**
3. **哈希表基准测试显示了最大差距**（与 Python 3 相比 183,000 倍），这是由于缺少原生哈希表
4. **H# 在所有基准测试中都产生了正确结果**，证明了语言的正确性

### 性能背景：

这些结果既不令人惊讶也不令人沮丧。H# 不是为与 C 或 Java 在执行速度上竞争而设计的。它被设计为一个优先考虑语言引导、教育清晰性和清晰语义设计的自托管语言。

最 impactful 的优化——实现原生哈希表——相对简单，可以弥补最大的性能差距。结合现有的自托管字节码编译器，H# 有一条清晰的路径来实现竞争性性能，同时不牺牲其作为自托管、从头开始的语言的独特特性。

### 未来工作：

1. 实现原生哈希表优化
2. 将字节码管道集成到解释器的主要执行路径中
3. 添加可变字符串构建器
4. 扩展基准测试套件，包括 I/O 密集型工作负载、并发模式和 GUI 渲染
5. 进行后续基准测试研究以量化改进

### 斐波那契(30) 调用计数

对 `fib(n)` 的调用次数遵循斐波那契数列：

```
fib(0) = 1 次调用
fib(1) = 1 次调用
fib(2) = 3 次调用
fib(3) = 5 次调用
fib(4) = 9 次调用
...
fib(30) = 2,692,537 次调用
```

### 哈希表复杂度分析

H# 使用线性搜索：每次查找 O(n)
- 5,000 次插入 + 5,000 次查找
- 每次查找扫描最多 5,000 个条目
- 总比较次数：5,000 × 5,000 = 25,000,000（最坏情况）
- 平均情况：5,000 × 2,500 = 12,500,000 次比较

Python 3 使用哈希表：每次操作 O(1)
- 5,000 次插入 + 5,000 次查找
- 总操作数：约 10,000

性能比：12,500,000 / 10,000 = 1,250×（理论）
实测比：183,183×（实际，包括常数因子）
